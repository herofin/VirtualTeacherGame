SECTION 1: Introduction

     This test effort will have testers running our program to find any bugs or program crashing errors that we missed in our initial coding of the program.
In addition to that we will hopefully find any design flaws that we didn't anticipate in our initial workup.

Reference Documents:

  • Concept of Operations
  • Project Management Plan
  • Software Requirements Specification

 

SECTION 2: Description of Test Environment

     The program will be written in Java and on Windows based machines. The testers will be mainly users but as the developers, we will be testing whenever a user isn't available. The testing environment wont be the same environment that the software will operate in. The program is geared towards grade school kids but because we don't know any high school children, we are going to test it with our peers. We are confident that this will still be still be an effective form of testing.

SECTION 3: Stopping Criteria

     We will test our program for as long as we can before reaching a point where we have to begin finalizing the product in order to meet the deadline. During a typical testing session we will have the user go through the program for 30 mins to an hour testing all sorts of conditions. Within that time frame we will observe and write down any bugs we notice without stopping the testing process. In case of a program crashing bug, we will restart the software, write down what may have triggered that crash and tell the user to avoid whatever caused it.
     In the unlikely case where no errors are found we will still continue to test every possible scenario within our program.
     Good enough to deliver will be defined as we have eliminated the largest software breaking bugs and the game is relatively bug free. Any remaining bugs will be minor and wont hamper the users experience.

 

SECTION 4: Description of Individual Test Cases

Describe EACH individual test to be run:

  • Test Objective: Test that the program responds as it should
  • Test Description: A quick test that makes sure the interactions on the screen are responsive to the players mouse clicks and keyboard. This will test our programs actionListeners and mouselisteners are working correctly.
  • Test Conditions: See Test Environment.
  • Expected Results: Buttons should respond when clicked and text fields should respond to keyboard presses.
  • Test Objective: Test the user account management
  • Test Description: Have the user create, delete, save, and rename their account. This information will be stored in a User account object.
  • Test Conditions: See Test Environment.
  • Expected Results: The User's account information should successful ly be stored in an user account object, and should be able to be modified when needed.
  • Test Objective: Test all correct answers within a day.
  • Test Description: The user will be given the correct answers and they will see that the program behaves accordingly when someone gets a 100% on their tests. We will be testing when question objects are answered correctly, their correctAnswer() methods are working properly.
  • Test Conditions: See Test Environment.
  • Expected Results: The program should display their 100%/A+ score at the end of the day and store this information in a week object that keeps track of all of that week's scores.
  • Test Objective: Test a mix of correct and wrong answers
  • Test Description: The user will purposefully get a mix of answers correct and wrong. This will test our daily scores algorithm and see if it gives the user a passing grade or not .
  • Test Conditions: See Test Environment.
  • Expected Results: If the user is below passing, <70%, it should tell the user and display a retry button. If the user has a passing grade, a button should be displayed that allows them to move on to the next day.
  • Test Objective: Test the end of week results display
  • Test Description: Have the user finish a week and have the end of week results get displayed onto the screen, and see if the user can now move on to the next week. The user may also be able to go back and improve their grades.
  • Test Conditions: See Test Environment.
  • Expected Results: At the end of the week the screen should display their stats for that week. Telling the user what subjects they were weak in and which ones were their strongest. The user should be able to go back and retake a course to improve their grades if they wanted to and move on to a new week with all new lectures and questions.
  • Test Objective: Test the end of school year report
  • Test Description: Have the user finish the school year and see how our program reacts to that scenario. This will test the interaction between week objects and their sequencing.
  • Test Conditions: See Test Environment.
  • Expected Results: A congratulatory message should be displayed to the screen, along with options to see the user's stats through the subjects. Afterwards the user will be redirected to the menu.
