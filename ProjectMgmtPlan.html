Project Overview

This project is an educational game meant for users around the ages of 5 to 15. By playing the game, users will learn and reinforce material that is common in school curriculums. The game's setting will be a virtual school, and the player will be a student at the school. The gameplay will be sectioned off into school weeks (Monday through Friday). Each day, the player will attend 5 periods. Periods 1, 2, 4, and 5 will be courses Math, Language Arts, Science, and History, though not necessarily in that particular order. Period 3 will be lunch, and this is where the player might receive encouragement or constructive criticism from his/her virtual peers. After the school day, the player goes home and can do homework. The homework will be how the player gets feedback on how they are doing in their courses. At the end of a week, the player will be presented with a view of how they did throughout the week. After a certain number of school weeks, the player will have finished a year of school and can advance to the next year.

Project Team Organization

Our group consists of 6 hardworking and dedicated members: Jason Shih , Jason Shipman, Marlon Smith, Anthony Delprete, Stephen Caskey, and Kevin Kruetek. We shall appoint a project manager only if we as a group decide that our current plan is not working or is not producing desirable results. We will delegate a specific document of each deliverable to either a single person or a mini team within the group to facilitate efficiency and communication.

One person will be responsible for scheduling face-to-face meetings outside of our pre-scheduled weekly meetings every friday. Our main form of communication is through Google Talk since all of our members have smartphones that are capable of this form of instant messaging. Our secondary forms of communication are text messaging, email and phone calls (in the event of an emergency). We have also decided to do textual editing of any documents inside Google Docs then any final drafts will be deposited into our code and document repository (as HTML) using GItHub.

Software Life Cycle Process

We will be following the Phased Incremental Developmental Process. This process is defined as starting with a small functional subsystem and adding functionality with each new release. We chose this process because we feel that it will fit our develop plan very well. We plan to design a learning/mentoring game whose main subjects are math, language arts, science, and history. As we get further into developing the critical learning portion of the game, hopefully mini games and more features will be added to fill out the experience and provide users with a greater variety of activities.

Configuration Management

For version control, we are using git. All code and documents are hosted on a Github repository. Jason Shipman is ultimately responsible for managing the repository, though every team member will use git for their individual additions and modifications.

Team members will fork Jason Shipman’s stable branch. This will provide each team member with their own copy of the repository which they are free to modify. When a team member is satisfied with their changes for a particular task and they have ensured none of their changes conflict with Jason Shipman’s stable branch, the team member will issue a pull request on Github to Jason Shipman. Jason Shipman will then merge the team member’s changes into his stable branch. At this point, all other team members can update their copies of the repository to the newest version.

When a team member makes modifications to their copy of the repository, the steps are as follows:

  1. Add/modify/delete source file(s).
  2. Include these changes for the next commit.
  3. Commit these changes.
  4. Update copy of repository to Jason Shipman's stable repository's latest version.
  5. Ensure changes made do not conflict with the stable repository. (If conflicts occur, fix them!)
  6. Repeat steps 1-5 if multiple issues are addressed across multiple commits.
  7. Issue a pull request on Github for Jason Shipman to consider merging team member’s changes into the stable repository.

Quality Assurance

Our Quality Assurance plan consists of tracking changes to the following:

  • Code and Inspections: inspections and changes to code will be made on a weekly basis based on the individual logs of members who have worked on the source code since the last inspection. These inspections will predominantly be made through comments on Github when a pull request is issued.
  • Design: if requirements or design plans change during development we shall update the respective documents.
  • Test Plan: we will try to make as few changes to our test plan as possible outside of adding more efficient tests and rigorous tests.

Risk Management

  • Loss of a Member: Reassign responsibilities efficiently through the remaining group members.
  • Missing Deadlines: We will check in with everyone’s progress via email every two days to make sure that we as a group are on track.
  • Data Loss We plan to use an online repository and maintain offline backups for our documents and code

Plan for tracking, control, and reporting of progress

At a minimum, each team member will post the following information weekly: individual time and activity log, individual status information, individual issues and problems, and individual defect log. Each week, the project manager will: read and analyze the logs; examine the technical content of the work done to date; examine the technical progress metrics; consider the QA results; reassess the potential project risks; and take corrective action if necessary.

The project manager will issue a Project Management Report on the schedule as indicated in the deliverables section above. At a minimum, the Project Management Report will be generated every two weeks and will include the following information: 1 sentence description of overall status, 1 or 2 sentence of any planned changes to the project plan, graph of planned vs actual time, graph of planned vs actual for each technical progress metric, updated PERT chart, and anything else he/she deems important.
